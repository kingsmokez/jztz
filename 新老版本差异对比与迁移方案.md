# jztz_v17 新老版本差异对比与迁移方案（修订版）

> 对比基准：
> - **旧版**：`web_app.py.bak`（227 549 字节，约 5 477+ 行，单体 Flask 应用）
> - **新版**：`web_app.py`（7 050 字节）+ `routes/`（6 个 Blueprint）+ `modules/`（16 个业务模块）
>
> 对比时间：2026-06-01（修订）
>
> ⚠️ **修订说明**：初版文档结论"新版已包含旧版全部功能"不准确。经逐行代码对比，发现多个关键功能在新版中存在实现差异或缺失，直接影响选股结果的准确性。

---

## 1. 总体结论（修订）

**新版并非旧版的完全超集**。虽然新版在架构、安全性、新增功能（AI/Dexter/强势选股等）方面优于旧版，但在**核心选股逻辑**上存在以下关键差异：

| 严重程度 | 差异项 | 影响 |
|---------|--------|------|
| 🔴 严重 | 行业PE动态调整失效 | 价值因子评分不准确，高成长行业股票被低估 |
| 🔴 严重 | `run_picker` 缺少板块轮动和热点因子 | 每日选股结果缺少热点关联，与旧版结果差异大 |
| 🟡 中等 | `get_stock_industry` 数据源和映射逻辑不同 | 行业分类可能不一致 |
| 🟡 中等 | 竞价确认数量不同（3只 vs 5只） | 竞价选股结果数量不一致 |
| 🟡 中等 | 定时任务机制不同 | 新版调度器不是固定时间触发 |
| 🟢 轻微 | 板块行情数据源不同 | 板块数据字段可能有差异 |

## 2. 关键差异详解

### 2.1 🔴 行业PE动态调整失效（最严重）

**旧版实现**（`web_app.py.bak:948-1007`）：

```python
def get_stock_industry(code):
    # 1. 调用 emweb.eastmoney.com/PC_HSF10/CompanySurvey/CompanySurveyAjax 获取行业
    # 2. 将行业名匹配 SECTOR_PE_RANGES 字典
    # 3. 返回 pe_fair_max 和 pe_fair_low
    result = {
        'industry': industry,
        'sector_type': sector_type,
        'pe_fair_max': sector_config.get('pe_fair_max', 30),  # 如半导体=100
        'pe_fair_low': sector_config.get('pe_fair_low', 15),   # 如半导体=28
    }
```

**新版实现**（`modules/data_fetcher.py:438-465`）：

```python
def get_stock_industry(code):
    # 1. 调用 RPT_LICO_FN_CPD API 获取 EM_INDUSTRY
    # 2. 直接返回，没有匹配 SECTOR_PE_RANGES
    result = {"industry": industry, "sector_type": "default"}
    # ❌ 缺少 pe_fair_max 和 pe_fair_low 字段！
```

**影响链路**：

```
get_stock_industry() → mf_score_value() → multi_factor_evaluate() → evaluate_stock() → run_picker()
```

- `mf_score_value`（`modules/scoring.py:545`）调用 `get_stock_industry(code)` 获取 `pe_fair_max` 和 `pe_fair_low`
- 新版始终返回默认值 `pe_fair_max=30, pe_fair_low=15`
- **结果**：半导体行业PE=80的股票，旧版认为合理（pe_fair_max=100），新版认为高估（pe_fair_max=30），价值因子评分严重偏低

**修复方案**：修改 `modules/data_fetcher.py` 的 `get_stock_industry` 函数，增加 `SECTOR_PE_RANGES` 匹配逻辑，返回 `pe_fair_max` 和 `pe_fair_low`。

### 2.2 🔴 `run_picker` 缺少板块轮动和热点因子

**旧版 `run_picker`**（`web_app.py.bak:2354-2748`）完整流程：

1. 获取板块轮动策略（`get_daily_priority_sectors`）
2. 获取热门板块成分股（`get_hot_stocks_from_sectors`）
3. 获取全市场实时行情
4. 合并热门股票到候选池
5. 基础过滤（换手率、ST等）
6. 并发校准全部财务数据（30线程）
7. 计算技术指标（15线程）
8. 五维评估 + 热点因子计算（`calculate_hot_factor`）
9. 动态排序（v5评分 + 涨跌幅调整）
10. 热门股票占比控制（≤30%）
11. 返回 Top 50

**新版 `run_picker`**（`modules/stock_picker.py:14-187`）简化流程：

1. 获取全市场行情
2. 基础过滤
3. 批量获取财务数据
4. 补充PB
5. 批量计算技术指标
6. 五维评分
7. 按 v5_score 排序返回 Top 50

**缺失功能**：

| 功能 | 旧版 | 新版 | 影响 |
|------|------|------|------|
| 板块轮动策略 | ✅ `get_daily_priority_sectors` | ❌ 缺失 | 无法根据当日热门板块加分 |
| 热门股票合并 | ✅ `get_hot_stocks_from_sectors` | ❌ 缺失 | 热门板块股票不会自动加入候选 |
| 热点因子计算 | ✅ `calculate_hot_factor` | ❌ 缺失 | 股票无热点加分，评分偏低 |
| 涨跌幅动态调整 | ✅ `change_adj` | ❌ 缺失 | 评分无当日行情调整 |
| 热门股票占比控制 | ✅ ≤30% | ❌ 缺失 | 可能热门股占比过高 |
| 离线模式fallback | ✅ 预设库兜底 | ❌ 缺失 | API失败时无数据 |

**修复方案**：在 `modules/stock_picker.py` 中补充板块轮动策略获取、热门股票合并、热点因子计算等逻辑，或在 `modules/` 下新建 `hot_factor.py` 模块。

### 2.5 🟡 定时任务机制不同

**旧版**（`web_app.py.bak:556-617`）：
```python
def schedule_daily_pick():
    while True:
        current_time = now.strftime("%H:%M")
        if current_time == "09:27":  # 固定时间触发
            execute_daily_pick('morning')
        elif current_time == "14:30":  # 固定时间触发
```

**新版**（`web_app.py:155-163`）：
```python
scheduler.add_job("daily_picker", run_daily_picker, interval_seconds=300)  # 每5分钟
```

## 5. 具体修复代码指引

### 5.1 修复 `get_stock_industry`（最关键）

**文件**：`modules/data_fetcher.py`

**修改位置**：`get_stock_industry` 函数（约第438行）

**修改方案**：在获取行业名后，增加 SECTOR_PE_RANGES 匹配逻辑：

```python
def get_stock_industry(code: str) -> dict:
    # ... 现有获取行业名的逻辑 ...
    
    # === 新增：匹配 SECTOR_PE_RANGES ===
    from modules.scoring import SECTOR_PE_RANGES
    
    sector_type = 'default'
    pe_fair_max = 30
    pe_fair_low = 15
    
    for sname, sconfig in SECTOR_PE_RANGES.items():
        # 优先匹配 industry_names
        if 'industry_names' in sconfig:
            for ind_name in sconfig['industry_names']:
                if ind_name in industry or industry in ind_name:
                    sector_type = sname
                    break
        # 其次匹配 keywords
        if sector_type == 'default' and 'keywords' in sconfig:
            for kw in sconfig['keywords']:
                if kw in industry:
                    sector_type = sname
                    break
        if sector_type != 'default':
            break
    
    if sector_type != 'default':
        sector_config = SECTOR_PE_RANGES[sector_type]
        pe_fair_max = sector_config.get('pe_fair_max', 30)
        pe_fair_low = sector_config.get('pe_fair_low', 15)
    
    result = {
        "industry": industry,
        "sector_type": sector_type,
        "pe_fair_max": pe_fair_max,
        "pe_fair_low": pe_fair_low,
    }
    _industry_cache[code] = result
    _industry_cache_time = now
    return result
```

| P1 | 修复定时任务为固定时间触发 | 选股在正确时段执行 |

## 9. 风险点

1. **`get_stock_industry` 修改可能影响其他调用方**：需确认所有调用 `get_stock_industry` 的地方都能正确处理新增的 `pe_fair_max` 和 `pe_fair_low` 字段。
2. **`run_picker` 补充逻辑后性能影响**：板块轮动和热点因子计算需要额外API调用，可能增加选股耗时。
3. **定时任务修改**：从间隔轮询改为固定时间触发，需确保 `routes/daily.py` 的 `_schedule_daily_pick` 与 `web_app.py` 的调度器不冲突。
4. **SSE 持续推送**（`web_app.py:58`）：`while True` 循环，测试环境下需手动断连。
5. **Dexter 子系统**依赖外部 LLM，无 LLM key 时返回"模块暂未启用"。
